# 설계 결정

## D-LOG-01: JSONL 포맷 채택

**문제**: 시스템 로그를 자동으로 분석·집계할 주체(감시 에이전트, 외부 뷰어)가 멀티라인 평문 포맷을 재파싱해야 한다면 비용이 크고 경계가 모호하다.

**결정**: 공식 파일 포맷으로 **JSON Lines**(한 줄에 한 JSON 객체)를 사용한다.

**근거**:
- 1라인 = 1이벤트 → 스택 트레이스의 멀티라인 경계 모호함이 사라진다.
- 필드 구조화 → `level=ERROR`, `account_id=paper01` 같은 조건 집계가 자연스럽다.
- 표준 라이브러리 `json`만으로 파싱·생성 가능하다 — 외부 의존성 없음.
- grep 호환성도 유지된다 (평문 대비 손해 없음).

## D-LOG-02: 이중 핸들러 — stdout 평문 / 파일 JSONL

**문제**: JSONL은 기계 친화적이지만 사람이 직접 읽기에 불편하다. 반대로 평문만 유지하면 자동 분석이 어렵다.

**결정**: 두 핸들러를 병행한다.

| 핸들러 | 대상 | 포맷 | 용도 |
|---|---|---|---|
| `StreamHandler(sys.stdout)` | 콘솔 | 평문 (`%(asctime)s [%(levelname)s] %(name)s: %(message)s`) | 사람이 직접 관찰 |
| `TimedRotatingFileHandler` | `logs/ante-YYYY-MM-DD.jsonl` | JSONL | 에이전트·스크립트 분석 |

**근거**: 두 핸들러가 동일한 로그 레코드를 각자의 포맷으로 기록한다. 한쪽이 실패해도 다른 쪽은 유지된다. 개발자 경험과 자동 분석을 모두 만족한다.

## D-LOG-03: `ANTE_LOG_JSONL` 환경변수 게이트

**문제**: JSONL 파일 핸들러를 전 환경에 일괄 도입하면 회귀 위험이 있다.

**결정**: 파일 핸들러는 환경변수 `ANTE_LOG_JSONL=1`일 때만 활성화한다. 미설정(기본) 시 stdout 평문 핸들러만 동작한다.

**근거**: 스테이징·QA에서 먼저 사용해 안정성을 확인한 뒤 프로덕션에 점진 도입한다. 기존 운영 동작과 완전 호환되며 회귀 위험이 0이다.

## D-LOG-04: `ANTE_ENV` 환경 식별

**문제**: 다중 환경이 동일한 이슈 추적 시스템·분석 도구를 공유할 때 환경 구분이 필수이다.

**결정**: 로그 레코드의 `env` 필드를 환경변수 `ANTE_ENV`에서 주입한다. 기본값은 `production`, 스테이징은 `staging`, QA는 `qa`.

**근거**: 필드로 분리하여 후속 쿼리·필터링을 단순화한다. 메시지에 섞어 넣으면 파싱 비용과 실수가 동반된다.

## D-LOG-05: 표준 라이브러리만 사용

**문제**: `structlog`, `python-json-logger` 등 외부 라이브러리가 JSON 로깅을 쉽게 해준다. 그러나 의존성을 추가하면 버전 관리·보안 업데이트 부담이 생긴다.

**결정**: `logging.Formatter` 상속과 표준 `json` 모듈만으로 구현한다.

**근거**:
- 현재 30+ 모듈이 `logging.getLogger(__name__)` 표준 패턴을 쓰고 있어 호출부 수정이 전혀 필요 없다.
- JSON 직렬화 로직은 80라인 내외로 직접 작성 가능하다.
- 외부 의존성 최소화는 [AGENTS.md](../../../AGENTS.md)의 설계 철학에 부합한다 (얇은 인프라).
