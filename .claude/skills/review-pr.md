# PR 코드 리뷰 스킬

> PR에 대해 구조화된 체크리스트 기반 코드 리뷰를 수행하고, 결과를 PR 코멘트로 게시한다.

## 트리거

- `/review-pr #{PR번호}` — 오케스트레이터 또는 사용자가 수동 실행
- `/implement-issue` 12단계에서 CI 통과 후 자동 실행

## 인자

$ARGUMENTS — PR 번호 (예: 42)

## 실행 절차

### 1단계: PR 정보 수집

```bash
# PR 메타데이터
gh pr view #{PR번호} --json title,body,baseRefName,headRefName,labels,files

# 변경된 파일 목록
gh pr diff #{PR번호} --name-only

# 전체 diff
gh pr diff #{PR번호}
```

연관 이슈 번호를 PR 본문에서 추출한다 ("Closes #N" 패턴).

### 2단계: 컨텍스트 로딩

PR의 변경 영역에 따라 필요한 문서를 읽는다:

| 변경 영역 | 로딩 대상 |
|-----------|----------|
| `src/ante/web/` | `docs/specs/web-api.md`, `.claude/skills/module-conventions.md` |
| `src/ante/{모듈}/` | `docs/specs/{모듈}.md`, `.claude/skills/module-conventions.md` |
| `frontend/` | `docs/dashboard/architecture.md`, `.claude/skills/frontend-conventions.md` |
| `tests/` | `docs/runbooks/04-testing.md` |
| async 코드 포함 | `.claude/skills/asyncio-patterns.md` |
| DB 코드 포함 | `.claude/skills/sqlite-patterns.md` |

연관 이슈의 유저스토리와 수용 조건도 읽는다:
```bash
gh issue view #{이슈번호}
```

### 3단계: 체크리스트 검증

아래 체크리스트를 **순서대로** 검증한다. 각 항목은 PASS / FAIL / N/A 중 하나로 판정한다.

---

#### A. 이슈 정합성

| # | 검증 항목 | 판정 기준 |
|---|----------|----------|
| A1 | 유저스토리 충족 | 이슈의 모든 US가 구현되었는가 |
| A2 | 수용 조건 충족 | 이슈의 완료 조건이 모두 만족되었는가 |
| A3 | 범위 초과 변경 없음 | 이슈에 명시되지 않은 변경이 포함되어 있지 않은가 |

#### B. API 계약 (#386 관련)

| # | 검증 항목 | 판정 기준 |
|---|----------|----------|
| B1 | response_model 존재 | 새 엔드포인트에 `response_model`이 명시되어 있는가 |
| B2 | 스키마 무단 변경 없음 | `schemas.py`의 기존 Pydantic 모델이 변경되었는가 → 변경 시 별도 이슈 필요 |
| B3 | 프론트엔드 타입 동기화 | API 응답 변경이 있다면 프론트엔드 타입도 함께 갱신되었는가 |

> B 섹션은 #386 에픽 완료 후 활성화. 그 전까지는 전체 N/A 처리.

#### C. 설계 문서 정합성

| # | 검증 항목 | 판정 기준 |
|---|----------|----------|
| C1 | 스펙 문서 일치 | 변경 내용이 `docs/specs/{모듈}.md`와 일치하는가. 스펙이 SSOT이므로 불일치 시 FAIL 처리하고 별도 이슈로 해결 권고. 리뷰 중 스펙을 수정하지 않는다 |
| C2 | 생성 문서 갱신 | DDL 변경 → `db-schema.md`, API 변경 → `api-reference.md`, CLI 변경 → `cli-reference.md` 갱신 여부 |

#### D. 코드 품질

| # | 검증 항목 | 판정 기준 |
|---|----------|----------|
| D1 | 컨벤션 준수 | 대상 영역의 skills/ 컨벤션을 따르는가 (모듈 구조, DI, 타입 힌트 등) |
| D2 | 사이드이펙트 | 변경이 다른 모듈에 의도하지 않은 영향을 주는가 (공유 import, 공유 스키마 등) |
| D3 | 에러 처리 | 새 코드의 에러 처리가 적절한가 (모듈별 예외 계층 사용, 적절한 HTTP 상태 코드) |
| D4 | 보안 | SQL 인젝션, XSS, 인증 우회 등 OWASP 취약점이 없는가 |

#### E. 테스트

| # | 검증 항목 | 판정 기준 |
|---|----------|----------|
| E1 | 테스트 존재 | 새 기능/수정에 대응하는 테스트가 있는가 |
| E2 | 엣지 케이스 | 경계값, 에러 경로 등 엣지 케이스가 커버되는가 |
| E3 | 기존 테스트 영향 | 기존 테스트가 불필요하게 수정/삭제되지 않았는가 |

#### F. PR 형식

| # | 검증 항목 | 판정 기준 |
|---|----------|----------|
| F1 | 커밋 메시지 | Conventional Commits 형식 준수 (`feat(scope): ...`) |
| F2 | PR 크기 | 변경 라인 수 300줄 이하 (테스트 제외). 500줄 초과 시 분할 권고 |
| F3 | base 브랜치 | 에픽 소속이면 에픽 브랜치, 독립이면 main이 base인가 |

### 4단계: 판정

체크리스트 결과를 종합하여 최종 판정을 내린다:

| 판정 | 조건 |
|------|------|
| **APPROVE** | FAIL 항목 0개 |
| **REQUEST_CHANGES** | FAIL 항목 1개 이상 |

### 5단계: PR 코멘트 게시

판정 결과를 PR 코멘트로 게시한다. 포맷:

```markdown
## Code Review — {APPROVE | REQUEST_CHANGES}

> PR #{PR번호} | 이슈 #{이슈번호} | 리뷰 일시: {YYYY-MM-DD HH:MM}

### A. 이슈 정합성
- [x] A1 유저스토리 충족
- [x] A2 수용 조건 충족
- [ ] A3 범위 초과 변경 없음 — ⚠️ {설명}

### B. API 계약
- N/A (#386 완료 전)

### C. 설계 문서 정합성
- [x] C1 스펙 문서 일치
- [x] C2 생성 문서 갱신

### D. 코드 품질
- [x] D1 컨벤션 준수
- [x] D2 사이드이펙트 없음
- [x] D3 에러 처리 적절
- [x] D4 보안 취약점 없음

### E. 테스트
- [x] E1 테스트 존재
- [ ] E2 엣지 케이스 — ⚠️ {설명}
- [x] E3 기존 테스트 무영향

### F. PR 형식
- [x] F1 커밋 메시지
- [x] F2 PR 크기
- [x] F3 base 브랜치

---

### 수정 요청 (REQUEST_CHANGES인 경우)

1. **A3**: {구체적 수정 사항과 이유}
2. **E2**: {구체적 수정 사항과 이유}

---

🤖 Reviewed by Claude Code (`/review-pr`)
```

코멘트 게시:
```bash
gh pr comment #{PR번호} --body "{위 내용}"
```

### 6단계: 수정 루프 (REQUEST_CHANGES인 경우)

1. FAIL 항목에 대한 수정 사항을 구현한다.
   - 수정은 **새 커밋**으로 추가한다 (amend 금지).
2. 수정 완료 후 푸시하고, 3단계부터 재실행한다.
3. **최대 2회** 수정 루프를 반복한다.
4. 2회 수정 후에도 FAIL이 남으면 사용자에게 에스컬레이션한다:
   ```
   [리뷰 에스컬레이션]
   - PR: #{PR번호}
   - 잔여 FAIL 항목: {목록}
   - 시도한 수정: {1차, 2차 각각}
   - 판단 요청: {사용자에게 결정을 요청하는 사항}
   ```

### 7단계: 머지

- **자동 머지 (야간 모드)**: APPROVE 판정 + CI 전체 통과 시 자동 squash merge.
- **수동 머지 (일반 모드)**: APPROVE 판정 결과를 사용자에게 보고하고 머지 승인을 요청한다.

```bash
# 사용자 승인 후
gh pr merge #{PR번호} --squash --delete-branch
```

## 판정 원칙

1. **객관적 기준만 적용**: "더 좋은 방법이 있을 것 같다" 같은 주관적 의견은 FAIL 사유가 되지 않는다. 체크리스트에 정의된 기준만으로 판정한다.
2. **현재 이슈 범위만 검증**: 기존 코드의 문제를 이 PR에서 고치라고 요구하지 않는다. 발견 시 별도 이슈로 등록한다.
3. **자동화 가능 항목 우선**: D4(보안), F1(커밋 형식), F2(PR 크기) 등은 가능한 한 도구로 자동 검증한다.
