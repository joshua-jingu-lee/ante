오픈 이슈를 아키텍트 관점에서 순회하며, 구현 시 주의사항과 보완 의견을 GitHub 코멘트로 남긴다.

## 인자

$ARGUMENTS — 옵션 (생략 가능)
- 없음: 전체 오픈 이슈 대상 (`feature`, `bug`, `refactor`, `epic` 라벨)
- `--label {라벨}`: 특정 라벨의 이슈만 대상
- `#{번호}`: 단일 이슈만 대상

## 목적

구현 에이전트가 이슈를 받아 작업하기 전에, 아키텍트 관점의 사전 검토를 제공한다.
이를 통해 구현 방향 이탈, 스펙 불일치, 모듈 간 의존성 충돌 등을 사전에 방지한다.

## 실행 절차

### 1단계: 이슈 목록 수집

```bash
# 전체 모드
gh issue list --state open --label "feature,bug,refactor,epic" --limit 50 --json number,title,labels,body

# 단일 이슈 모드
gh issue view #{번호} --json number,title,labels,body
```

이미 아키텍트 리뷰가 달린 이슈는 건너뛴다:
```bash
# 코멘트에 "🏗️ **아키텍트 리뷰**" 가 있으면 스킵
gh issue view #{번호} --json comments --jq '.comments[].body' | rg -q "🏗️ \\*\\*아키텍트 리뷰\\*\\*"
```

### 2단계: 이슈별 분석

각 이슈에 대해 다음을 순서대로 수행한다.

#### 2-1. 컨텍스트 로딩

이슈 본문에서 변경 대상 모듈을 식별하고, 관련 문서를 로딩한다:

| 식별 대상 | 로딩 문서 |
|-----------|----------|
| 이슈가 언급하는 모듈 | `docs/specs/{모듈}/{모듈}.md` |
| 아키텍처 전반 | `docs/architecture/README.md` (모듈 의존성 방향, EventBus 규칙) |
| 설계 결정 이력 | `docs/decisions/README.md` (관련 D-번호) |
| 기존 구현 | `src/ante/{모듈}/` 의 주요 파일 (서비스, 모델) |

#### 2-2. 검토 체크리스트

아래 항목을 순서대로 검토한다. 해당 없는 항목은 건너뛴다.

##### A. 스펙 정합성

| # | 검토 항목 |
|---|----------|
| A1 | 이슈가 요구하는 인터페이스/동작이 `docs/specs/`에 정의되어 있는가 |
| A2 | 스펙에 정의되어 있지만 이슈 본문이 스펙과 다른 부분이 있는가 |
| A3 | 스펙에 아직 미정의된 영역이 있어 구현 전 스펙 보강이 필요한가 |

##### B. 아키텍처 영향

| # | 검토 항목 |
|---|----------|
| B1 | 모듈 간 의존성 방향이 아키텍처 규칙을 위반하지 않는가 (순환 참조, 역방향 의존) |
| B2 | EventBus 이벤트 추가/변경이 필요한가 — 필요하다면 어떤 이벤트를 누가 발행/구독하는가 |
| B3 | DB 스키마 변경이 필요한가 — 마이그레이션 전략은 무엇인가 |
| B4 | API 스키마 변경이 필요한가 — 프론트엔드 타입 동기화 필요 여부 |

##### C. 구현 주의사항

| # | 검토 항목 |
|---|----------|
| C1 | 기존 코드에서 참고해야 할 유사 패턴이 있는가 (같은 모듈의 다른 기능, 비슷한 구조의 다른 모듈) |
| C2 | 성능 주의 지점이 있는가 (N100 저전력 환경, 벡터화 필요, 대량 데이터 처리) |
| C3 | 동시성 이슈가 있는가 (asyncio 패턴, 락 필요 여부, EventBus 핸들러 순서) |
| C4 | 에러 처리에서 특별히 고려할 케이스가 있는가 (네트워크 장애, 브로커 API 제한 등) |

##### D. 선행 조건 및 의존성

| # | 검토 항목 |
|---|----------|
| D1 | 이 이슈를 구현하기 전에 완료해야 할 다른 이슈가 있는가 |
| D2 | 이 이슈가 완료되면 차단이 풀리는 후속 이슈가 있는가 |
| D3 | API 스키마 변경이 선행되어야 하는가 (`docs/runbooks/01-development-process.md` §10) |

##### E. 에픽 이슈 전용 (epic 라벨인 경우)

| # | 검토 항목 |
|---|----------|
| E1 | 하위 이슈 간 의존성 순서가 적절한가 |
| E2 | 병렬 실행 가능한 하위 이슈 그룹이 올바르게 식별되었는가 |
| E3 | 하위 이슈 합산 범위가 에픽의 완료 조건을 충족하는가 (누락된 작업 없는가) |

### 3단계: 코멘트 작성 및 게시

분석 결과를 GitHub 이슈 코멘트로 게시한다.

```bash
gh issue comment #{이슈번호} --body "{코멘트}"
```

코멘트 포맷:

```markdown
🏗️ **아키텍트 리뷰**

### 관련 스펙
- `docs/specs/{모듈}/{모듈}.md` — {관련 섹션 요약}

### 주의사항
{검토 결과 중 구현 에이전트가 반드시 알아야 할 사항. 번호 매겨 기술.}

1. **{제목}**: {상세 설명. 코드 경로나 스펙 섹션 참조 포함}
2. ...

### 보완 권고
{이슈 본문 자체의 개선 제안이 있는 경우. 없으면 생략.}

- {보완 사항}

### 선행/후속 관계
{D1~D3 검토 결과. 해당 없으면 생략.}

---
🤖 Reviewed by Claude Code (`/arch-review`)
```

**작성 원칙:**

1. **실질적 가치만 작성**: "스펙을 잘 읽으세요" 같은 일반론은 쓰지 않는다. 구체적인 파일 경로, 함수명, 스펙 섹션 번호를 포함한다.
2. **간결함 우선**: 이슈당 코멘트는 주의사항 5개 이내로 압축한다. 모든 체크리스트 항목을 나열하지 않고, 실제 발견된 사항만 기술한다.
3. **판단 보류**: 스펙 불일치를 발견해도 스펙을 수정하지 않는다. 불일치 사실만 기록하고 사용자 판단을 요청한다.
4. **기존 코드 참조**: "이렇게 구현하라"보다 "기존 {모듈}의 {패턴}을 참고하라"로 안내한다.

### 4단계: 결과 요약

모든 이슈 순회 완료 후 사용자에게 요약을 보고한다:

```
## 아키텍트 리뷰 완료

| 이슈 | 제목 | 결과 |
|------|------|------|
| #101 | 계좌 CRUD | ⚠️ 주의사항 3건, 보완 권고 1건 |
| #102 | 봇 라이프사이클 | ✅ 특이사항 없음 |
| #103 | 리스크 엔진 | 🚫 스펙 미정의 — 구현 보류 권고 |

리뷰가 달린 이슈: {N}건
스킵 (이미 리뷰 완료): {M}건
```
